## Codebase Analysis

### What's Testable (by effort → reward ratio)

| Tier | Target | Why |
|------|--------|-----|
| **Easy** | DataAnnotations validation on `WebSiteConfiguration`, `LoginCredentials` | Pure attribute-based validation, zero mocking |
| **Easy** | Domain model constructors with null guards (`AspNetChannel`, `AspNetCoreReleaseInfo`, `AspNetRelease`) | Pure constructor logic |
| **Easy** | `WebSiteInstance.State`, `WebSiteRuntimeState` factories | Pure derived properties / static factories |
| **Easy** | All 16 Result types (`CreateSuccess`/`CreateFailure`) | Pure factory methods |
| **Easy** | `SemaphoreLock` (threading utility) | Single interface dependency (`ISemaphoreOwner`), no I/O |
| **Easy** | Extension methods (`ApiResponseExtensions`, `UriExtensions`) | Pure functions, no dependencies |
| **Medium** | `FileManagerService` (path sanitization) | Does real I/O — use temp directories |
| **Medium** | `ArchiveExtractorService` (zip-slip protection) | Mock `IFileManagerService`, real tar files in temp dirs |
| **Medium** | `VersionsDetectorService` (parsing logic) | Requires making private → `internal` + `[InternalsVisibleTo]` |
| **Medium** | `HttpClientExtensions` | Testable via `HttpMessageHandler` mock |
| **Hard** | `DsmApiClient` | No interface, real network, hardcoded config file read |
| **Hard** | `PlatformInfoService` | Constructor reads `appsettings.json` + `RuntimeInformation` |
| **Hard** | `DownloaderService` | External library (`Microsoft.Deployment.DotNet.Releases`) does network calls |
| **Defer** | `FileSystemService` (server), `WebSiteHostingService` | Complex, concrete `DsmApiClient` dependency |
| **Defer** | Blazor Components (Ui.Client) | Different paradigm (bUnit), defer |

## What's Deferred (and Why)

| Target | Blocking Factor | When to Address |
|--------|-----------------|-----------------|
| `DsmApiClient` | No interface, real network, hardcoded config file read | When extracting `IDsmApiClient` interface |
| `DownloaderService` | External library (`Microsoft.Deployment.DotNet.Releases`) does network calls | When we need download-specific coverage |
| `FileSystemService` (server) | Depends on concrete `DsmApiClient` | After `DsmApiClient` extraction |
| `WebSiteHostingService` | Complex orchestration, process spawning, channels | Integration test phase |
| Blazor Components (Ui.Client) | Different paradigm (bUnit), defer | UI testing phase |
| `ProcessTerminator` | Static, P/Invoke to `libc.kill` | Not practical to unit test |

### Deferred Services — Detailed Analysis

#### `DsmApiClient`

**Constructor is partially mockable** — `IHttpClientFactory` and `ILogger<DsmApiClient>` are both interfaces. However:

- `ReadSettingsAsync()` calls `File.ReadAllLinesAsync(SystemDefaults.ConfigurationFileName)` — a hardcoded path (`/etc.defaults/system-mods.conf` on Synology). No abstraction, no injection.
- `ConnectAsync()` chains `ReadSettingsAsync → HandShakeAsync → AuthenticateAsync`, all `private`, all hitting the network.
- Even with a mocked `IHttpClientFactory`, the config file read blocks the flow.

**Fix required:** Extract `ISystemConfigReader` (or `IDsmApiClient` interface with a testable config injection).

---

#### `DownloaderService`

**Constructor is fully mockable** — `IPlatformInfoService` and `IFileManagerService` are interfaces. However:

- All methods call `ProductCollection.GetAsync()` from `Microsoft.Deployment.DotNet.Releases` — a static entry point that hits the Microsoft CDN.
- `file.DownloadAsync(...)` is an extension method on `ProductReleaseFile` from the same library — performs real HTTP download to disk.
- The library has no interfaces; it's a concrete, network-bound dependency.

**Fix required:** Wrap the library behind an `IProductCatalogService` interface that returns `Product`/`ProductRelease` models.

---

#### `WebSiteHostingService`

**Constructor is fully mockable** — all 5 dependencies (`ILogger`, `ILoggerFactory`, `IWebSitesConfigurationService`, `IFileSystemService`, `IReverseProxyManagerService`) are interfaces. However:

- Internally creates `SiteLifecycleManager` via `new`, which spawns real processes.
- `StartAsync`/`StopAsync` exercise `SiteLifecycleManager` channels involving process lifecycle, SIGTERM, and `ProcessTerminator` P/Invoke.
- `AddWebsiteAsync` exercises a 4-step flow (permissions → reverse proxy → config persist → instance creation). Testing requires verifying 4 different mocked services interact correctly in sequence.

**Fix required:** Extract `IProcessRunner` from `SiteLifecycleManager`, or accept that this is an integration test rather than a unit test.

### Decision Points Resolved

1. **InternalsVisibleTo for Phase 8?** — ✅ Approved. Low-risk change, enables parsing tests in isolation.
2. **Real temp dirs vs. file system abstraction?** — Real temp dirs used throughout (Phases 6, 7, 9). No flakiness observed.
3. **Zip-slip test approach** — Pure C# `TarWriter.WriteEntry()` with malicious entry name (`../../escaped.txt`). No external tool dependency.
